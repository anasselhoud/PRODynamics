  • Item Version : v 0.1.0

This tool is part of a global solution that has been developed during the industrial PhD thesis at FORVIA Clean Mobility between February 2022 and December 2024. If you wanna add more features, this guide is for you!

PRODynamics is an optimization and simulation tool designed to evaluate the impact of hazardous events on a production line, as well as optimizing its material flow. It models the flow between various resources with high speed and accuracy, factoring in multiple events such as machine breakdowns, microstops, supply chain delays, and more.

This documentation is to help you regarding each step of developement. Please go through the documentation carefully to understand how this template is made and how to edit this properly. Basic algorithmic and programming knowledge is required to customize the tool. For a more theoretical view, refer to chapter V of my PhD dissertation.

How to Set Up

Create Conda Environement

Start by installing Miniconda, and create your environment based on Python 3.9. Refer to the official website of Conda to install it and set it up.

Install Managing Environments

Set Up Environement

Once the environement is created, it should be empty at first. In order to set up the environement with all libraries that we will need, I have prepared a "requirments.txt" file that contains list of all required Python libraries with their specific versions.

You can install the dependencies all at once, by activating the conda environement, going to the main folder placement, and use the command:

 conda activate "env_name" 
 pip install -r requirements.txt

Launch the Tool

We use streamlit framework to build and deploy the tool, it's fast and very easy to use. To run the full application locally, use this command on your conda prompt: 

streamlit run app.py

You got it right, my friend! The file "app.py" is the main script for the full app. We will get to that later when talking about the main folder architecture.

Streamlit offers the possibility to quickly deploy the tool online on the cloud for a better test of capabilities.

UML & Architecture

Let us dive deep now into the technical structure of the tool. As I explained before, the tool is composed of a simulation part, that models the flows of parts and material between automatic and manual resources, generate hazardous events and assess the performance of the production during the simulated time.

Simulation Part

The dynamic simulator was built using SimPy framework and organized in a modular architecture designed for flexibility and future expansion, as depicted in the provided diagram above. All simulation main functions are coded in the script: 

utils.py
The process begins with preparation, where users define the production line's characteristics. Here, users meticulously detail the production line itself, including the types and number of machines, the sequence of operations performed on each product, and the processing times involved. The definition of the product follows, where users specify the types of products, their bill of materials, and any assembly processes required. Finally, users can define stochastic events to introduce realism, such as machine breakdowns and product arrival rates. All the simulation models are therefore generated based on this data configuration. The simulation execution phase starts with multiple production models being initialized to explore various scenarios or configurations. A network representing the interconnected machines and buffers is then generated. Products travel between machines based on routing rules, transportation times, and buffer capacities. The core engine of the simulator tracks product progress through various operations, managing discrete events like product arrivals, machine operational completions, and breakdowns. Critically, data is collected throughout the simulation on key performance metrics such as throughput, lead time, work-in-process, and machine utilization.

Optimization Part

Up to the current version, the optimization section of the tool is composed of two main indepedent objectives: optimize the sizing of the intermediate buffers between machines - Optimize the transportation flow of materials between machines. All optimization main functions are coded in the script: 

utils_optim.py

Intermediate Buffer Sizing

For this part, we have developed a new method called SEOD (Sample-Estimate-Optimize-Decide), our surrogate model optimiza- tion method, that uses a simulation framework to solve the problem of buffers’ storage size optimization. Instead of perturbing the values like in Finite Perturbation Analysis, this method samples the values of possible buffer storage sizes following a specific distribution between the minimum and maximum storage size, evaluates the sample using the stochastic simulator, estimates the objective function and approximates it to a surrogate analytical model, and then optimizes it numerically to decide the optimal storage size for each buffer in the production line.

Transporation Flow Optimization

If intermediate buffer sizing can be managed using a surrogate model, optimizing transportation flow presents a more complex challenge. This problem is NP-hard and becomes even more intricate in real production environments. The goal is to regulate the movement of materials between machines while minimizing idle time. However, current transportation systems, such as robots or AGVs, follow pre-programmed, rigid instructions. In the event of a machine breakdown, the transporter continues to deliver materials to the malfunctioning machine, when it would be more efficient to skip it and supply another machine instead.
We attempt to solve the proble using Deep Reinforcement Learning (DRL), and more specifically Deep Q-Networks. The action space depends on the robot’s physical limitations and precedence constraints. The DRL agent determines control actions for the transporter, which could be a robot or an AGV. For simplicity, the action space is represented by a two-component vector (M-src, M-tgt), where M-src indicates the source machine from which a part is retrieved, and M-tgt indicates the target machine to which the part is delivered. With this formulation, the agent learns to generate a sequence of transportation actions based on the current state of the production line.
At each time-step, the agent uses the current state to decide what action to take to maximize future rewards. The state needs to contain enough information about the environment such that the agent is able to make the optimal decision. The states space is composed of a vector of production instant state metrics (following figure): state of machines (1 if operating, 0 otherwise), state of buffer (1 if full, 0 otherwise) and their current storage, the idle time of the machines, the current position of the transporter of the line. Simulation Classes

  • ManufLine - This is the main class of the simulation. It is composed of different attributes like: the SimPy environement, the simulation configurations (breakdown mode & law, simulation time...), the production line & process (number of machines, operators, robots/transporters, number of references, supply and outputs rates...)
  • Machine - This class represents the machine, it has as attributes: ID, processing time, buffer, linked machines (in and out), operational flag, loading flag, current reference in the machine...
  • Robot - This is the class representing the transporter, in general it is a manipulation/handling robot. Its main attributes are: transportation time, busy flag, sequence of transportation...
